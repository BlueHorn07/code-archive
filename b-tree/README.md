# B-Tree와 BST는 다르다.

- BST는 노드당 가질 수 있는 key가 1개임.
- B-Tree는 노드당 key가 여러 개 들어가고
  - 그 최대 개수를 결정하는 것이 `t` 계수임.
  - 노드 당 key 갯수는 `2t - 1`까지 가능함.
  - 그리고 노드 내에서는 key를 정렬된 상태로 저장함.

## 최대 child 갯수는 key 갯수 `k`보다 하나 더 만도록 설계 된다.

- BST 트리에서 자식 노드 수는 항상 left/right 2개 였음.
- B-Tree에서 key 갯수는 `2t-1`이 가능함.
- 만약 BTreeNode에 keys가 3개 였다면,

```
        [10 | 20 | 30]
       /    |    |    \
      c0   c1   c2   c3
```

이렇게 자식을 넣을 수 있음. 각 자식은 부모 key 기준으로 그 사이 범위에 있는 값을 저장함.

```
(-∞,10)
(10,20)
(20,30)
(30,+∞)
```

# 왜 keys가 3이 되면 선제적으로 split 하는가?

```
        [10 | 20 | 30]
       /    |    |    \
      5    15   25    35
```

이 상황에서 40을 삽입하면 둘 중 어떻게 되는가?

1번

```
          [20]
        /      \
      [10]     [30]
     /   \     /   \
   [5]  [15] [25] [35, 40]
```

2번

```
        [10 | 20 | 30]
       /    |    |    \
      5    15   25   [35|40]
```

1번, 2번 둘다 가능함.

1번은 top-down 방식으로 삽입 연산을 하기 위해 트리를 순회할 때마다, 항상 `if node.key is full: split node`를 먼저 수행함.

즉, full 상태의 노드는 삽입 연산을 위해 노드를 탐색하는 과정에서 모두 split 됨.

2번은 내가 실제로 삽입하려는 노드가 full 상태가 되면 split node를 수행함. 이 경우, split 하면서 child -> parent로 key를 올려주게 되는데, paraent가 꽉 차 있으면, parent를 한번더 split 해줘야 함.


```
              [40|80|120]
          /      |      |      \
 [10|20|30] [50|60|70] [90|100|110] [130]
```

이 상태에서 `5`를 insert 하게 되면,

```
              [40|80|120]
          /      |      |      \
 [5|10|20|30] [50|60|70] [90|100|110] [130]
 (overflow)
---
              [20 | 40 | 80 | 120]
               (overflow)
          /       |       |       |       \
     [5|10]     [30] [50|60|70] [90|100|110] [130]
    (20이 부모로 올라감.)
---
                  [80]
              /                \
          [20|40]                    [120]
         /   |   \                  /    \
   [5|10]  [30] [50|60|70] [90|100|110] [130]
```

2번 방식을 채택하면, split 후에 parent에서도 overflow가 발생해서 한번더 split이 발생할 수 있음.

즉, split이 연쇄적으로 전파됨.

하지만, 1번은 split이 연쇄적으로 발생하지는 않음. 상위 노드를 순회할 때 이미 full 상태인 노드를 정리하고 왔기 때문에, 중간에 split이 발생하더라도, split이 연쇄적으로 전파되지는 않음.
